# Runtime Code Cache

## Allocator Separation

Fact: MIR separates ordinary heap allocation from executable-code allocation.

`struct MIR_context` stores both `alloc` and `code_alloc` (`mir.c:31-37`). `_MIR_init` installs `default_alloc` when the caller passes `NULL` for heap allocation and `default_code_alloc` when the caller passes `NULL` for executable-code allocation (`mir.c:733-739`). Both pointers are then stored in the context (`mir.c:756-757`).

The executable-code allocator API is defined in `mir-code-alloc.h`:

- `MIR_code_alloc_t` is a pointer to `struct MIR_code_alloc` (`mir-code-alloc.h:21-26`).
- It contains callbacks for `mem_map`, `mem_unmap`, and `mem_protect` plus `user_data` (`mir-code-alloc.h:21-26`).
- Inline wrappers expose `MIR_mem_map`, `MIR_mem_unmap`, and `MIR_mem_protect` (`mir-code-alloc.h:28-38`).
- `MIR_mem_protect_t` has only two states: `PROT_WRITE_EXEC` and `PROT_READ_EXEC` (`mir-code-alloc.h:16-19`).

The custom allocator documentation says `MIR_alloc` is general purpose, while `MIR_code_alloc` is for executable-code pages and protection changes (`CUSTOM-ALLOCATORS.md:38-48`). It also says both allocator objects must live at least as long as the `MIR_context` (`CUSTOM-ALLOCATORS.md:34-36`, `CUSTOM-ALLOCATORS.md:52-54`).

Inference: Heap allocation owns MIR objects, vectors, strings, parser/generator/interpreter state, and temporary buffers. Code allocation owns page-level storage for thunks, wrappers, interpreter shims, and generated native code.

## Code Memory Lifecycle

Fact: Code memory is tracked in `struct machine_code_ctx`, allocated during context initialization and released during context finish.

Relevant structures in `mir.c`:

- `struct code_holder { uint8_t *start, *free, *bound; }` (`mir.c:4353-4355`).
- `struct machine_code_ctx { VARR(code_holder_t) *code_holders; size_t page_size; }` (`mir.c:4361-4364`).

`code_init(ctx)` allocates `machine_code_ctx` with the heap allocator, records the platform page size, and creates the code-holder vector (`mir.c:4492-4497`). `_MIR_init` calls `code_init` before creating the generator wrapper end and before interpreter initialization (`mir.c:786-789`).

`get_last_code_holder(ctx, size)` either reuses the final holder after 16-byte-aligning its `free` pointer, or maps a new page range through `MIR_mem_map(ctx->code_alloc, len)` (`mir.c:4369-4388`). The mapped region is pushed into `code_holders` with `start`, `free`, and `bound` pointers (`mir.c:4381-4388`).

Fact: `MAP_FAILED` is defined as `NULL` in the MIR code allocator header (`mir-code-alloc.h:14`), and `get_last_code_holder` treats that value as allocation failure (`mir.c:4381-4382`).

## Publication

Fact: Code publication means copying generated bytes into the current code holder and returning the executable address.

`_MIR_publish_code(ctx, code, code_len)` gets a holder large enough for `code_len`, calls `add_code`, and returns the allocated address (`mir.c:4426-4434`).

`add_code`:

- records the current `free` pointer as the destination address;
- advances `free` by `code_len`;
- builds a one-entry `MIR_code_reloc_t` where `offset = 0` and `value = code`;
- calls `_MIR_set_code` with `reloc_size = code_len`, which copies the bytes into the destination;
- flushes the instruction cache for the new range;
- returns the destination address (`mir.c:4412-4423`).

`_MIR_get_new_code_addr(ctx, size)` returns the current `free` pointer from a holder large enough for `size`, without advancing it (`mir.c:4486-4490`). `_MIR_publish_code_by_addr(ctx, addr, code, code_len)` publishes only if the current holder's `free` pointer still equals `addr` and there is enough space (`mir.c:4436-4444`).

Observed use: RISC-V64 `_MIR_get_wrapper` uses `_MIR_get_new_code_addr` and `_MIR_publish_code_by_addr` to build a wrapper whose PC-relative jump remains valid for the final address (`mir-riscv64.c:852-881`).

## Patching

Fact: Patches are represented either as raw byte replacement or as offset/value relocations.

`struct MIR_code_reloc` has:

- `size_t offset`
- `const void *value`

It is declared in `mir.h:687-692`.

Patch functions:

- `_MIR_change_code(ctx, addr, code, code_len)` copies `code_len` bytes to an existing code address and flushes that exact range (`mir.c:4446-4457`).
- `_MIR_update_code_arr(ctx, base, nloc, relocs)` writes pointer-sized relocation values at offsets from `base` and flushes from `base` through the largest updated pointer slot (`mir.c:4459-4469`).
- `_MIR_update_code(ctx, base, nloc, ...)` is a varargs wrapper that builds up to 19 relocation records and delegates to `_MIR_update_code_arr` (`mir.c:4472-4483`).

`_MIR_set_code` performs the actual protected write. If `reloc_size == 0`, it copies `sizeof(void *)` bytes from each relocation value into `base + offset`; otherwise it copies `reloc_size` bytes from each relocation value (`mir.c:4398-4408`).

Observed generated-code patching examples:

- x86-64 lazy BB thunk replacement rewrites the thunk into a relative jump via `_MIR_change_code` (`mir-x86_64.c:841-848`, documented earlier in `docs/07-lazy-basic-block-jit.md`).
- RISC-V64 whole-function or BB branch rebasing writes 8-byte absolute addresses or patches branch instructions through `_MIR_change_code` / `_MIR_update_code_arr` (`mir-gen-riscv64.c:2811-2822`, `mir-gen-riscv64.c:2918-2938`, `mir-gen-riscv64.c:2956-2967`).

## Protection Model

Fact: MIR's abstraction has two executable protection states: `PROT_WRITE_EXEC` and `PROT_READ_EXEC` (`mir-code-alloc.h:16-19`).

`_MIR_set_code` always calls:

1. `MIR_mem_protect(..., PROT_WRITE_EXEC)`
2. copy bytes or relocation values
3. `MIR_mem_protect(..., PROT_READ_EXEC)`

This is visible in `mir.c:4398-4409`.

Default non-Windows behavior:

- `PROT_READ_EXEC` maps to `PROT_READ | PROT_EXEC` (`mir-code-alloc-default.c:18-25`).
- `PROT_WRITE_EXEC` maps to `PROT_WRITE | PROT_EXEC`, except on RISC-V where it maps to `PROT_WRITE | PROT_READ | PROT_EXEC` (`mir-code-alloc-default.c:18-25`).
- Default `mem_map` uses `mmap` with `PROT_EXEC` on most Unix-like targets (`mir-code-alloc-default.c:58-64`).

Default Windows behavior:

- `PROT_WRITE_EXEC` maps to `PAGE_EXECUTE_READWRITE`.
- `PROT_READ_EXEC` maps to `PAGE_EXECUTE_READ`.
- `mem_map` uses `VirtualAlloc(..., MEM_COMMIT, PAGE_EXECUTE)` (`mir-code-alloc-default.c:66-82`).

Apple AArch64 special case:

- Default `mem_map` uses `MAP_JIT` with `PROT_EXEC | PROT_WRITE | PROT_READ` (`mir-code-alloc-default.c:58-61`).
- `default_mem_protect` uses `pthread_jit_write_protect_np(FALSE)` to allow writes and `pthread_jit_write_protect_np(TRUE)` plus `sys_icache_invalidate` for read/execute transitions when supported (`mir-code-alloc-default.c:28-51`).

Inference: MIR does not model strict W^X as "write-only then execute-only" in its public code allocator states. The normal states are executable in both write and read phases. On Apple AArch64, write protection is controlled through platform JIT write-protect APIs rather than a conventional `mprotect` path.

## Instruction Cache Synchronization

Fact: MIR has a generic `_MIR_flush_code_cache(start, bound)` helper. With GCC and not `__MIRC__`, it calls `__builtin___clear_cache(start, bound)` (`mir.c:4391-4395`).

Publication and patching call this helper:

- `add_code` flushes the newly published range (`mir.c:4412-4423`).
- `_MIR_change_code` flushes the changed range (`mir.c:4446-4457`).
- `_MIR_update_code_arr` flushes the updated pointer-relocation range (`mir.c:4459-4469`).

Apple AArch64 default protection also calls `sys_icache_invalidate` when transitioning to the read/execute state (`mir-code-alloc-default.c:40-43`).

Open question: On some platforms this may duplicate cache invalidation; on others the generic helper may compile to nothing if the compiler path is not available.

## Lifetime And Freeing

Fact: Code holders are released only as a group in `code_finish`.

`code_finish(ctx)` pops all code holders, calls `MIR_mem_unmap(ctx->code_alloc, ch.start, ch.bound - ch.start)` for each mapped region, destroys the holder vector, frees `machine_code_ctx`, and clears the context pointer (`mir.c:4499-4507`).

No inspected API frees an individual generated function or an individual thunk. Generated machine-code pointers such as `func_item->u.func->machine_code` are stored on function items, but their memory is owned by the context-level code holders and unmapped only during context teardown.

Inference: Individual compiled functions cannot be reclaimed independently in the observed runtime. This simplifies allocation and patching but couples generated code lifetime to the MIR context.

## Thread-Safety Assumptions

Fact: `_MIR_publish_code`, `_MIR_change_code`, `_MIR_update_code_arr`, and `_MIR_update_code` carry `thread safe` comments in `mir.c:4426-4472`.

Observed limitation: the inspected implementations mutate `code_holders`, holder `free` pointers, and code bytes without visible locks in `mir.c:4369-4489`. The whole-function generator also stores `func_item->u.func->machine_code` with a comment saying an atomic should be used, but C2MIR does not implement atomics yet (`mir-gen.c:9500-9501`).

Hypothesis: Thread-safety may rely on external synchronization, allocator-level synchronization, platform guarantees for code writes, or code paths not inspected yet. The visible implementation alone is not enough to prove safe concurrent generation or concurrent lazy thunk patching.

Custom allocator documentation also places thread-safety responsibility on callers who use custom allocators from multiple threads (`CUSTOM-ALLOCATORS.md:56-58`).

## Limitations And Trade-Offs

- Code allocation is page-holder based and context-lifetime scoped. This is simple, but does not support fine-grained freeing of generated functions.
- The protection abstraction has only write/execute and read/execute states. It does not express non-executable writable staging memory or execute-only code.
- The default Unix mapping starts with executable memory (`mir-code-alloc-default.c:58-64`), then toggles between executable states during writes.
- Code patching is central to thunks, wrappers, lazy generation, branch rebasing, and relocation. This complicates strict memory-protection environments.
- Thread-safety comments exist, but the inspected code does not show synchronization around code-holder mutation or function `machine_code` publication.
- Instruction-cache flushing is present, but platform coverage depends on compiler/platform support and default allocator behavior.

## Relevance To RISC-V32 / Fantasy Computer Extraction

- Heap allocator separation from executable code allocation: `required for native host execution`, useful but not essential to MIR semantics.
- Code holders, page mapping, protection toggles, and instruction-cache flushing: `required for native host execution`.
- Pointer-sized relocation records: `backend-specific engineering detail`; RISC-V32 would need 4-byte pointer-aware relocation sites where RV64 currently writes 8-byte addresses.
- Lazy thunk patching and branch redirection: `optimization convenience` and `backend-specific engineering detail`.
- Context-lifetime code cache: removable for a fantasy computer with fixed executable memory, a bytecode interpreter, or a static code arena.

In a fantasy computer with fixed executable memory, `MIR_code_alloc_t`, `MIR_mem_map`, `MIR_mem_unmap`, OS protection toggles, and OS cache flush hooks could be replaced by a simple arena or omitted entirely if code is interpreted. If native code is emitted into a real instruction memory, some target-specific cache synchronization may still be required.

## Open Questions

- Are code publication and patching intended to be called concurrently, or are the `thread safe` comments stale/incomplete?
- Are there platforms where `PROT_EXEC` without `PROT_READ` is intentionally required for initial mapping?
- Should a strict W^X allocator be possible with the current `PROT_WRITE_EXEC` / `PROT_READ_EXEC` enum?
- Can generated code be invalidated when a function is redefined, or is context teardown the only supported reclamation?
- Does Apple AArch64 cache invalidation happen both in `default_mem_protect` and `_MIR_flush_code_cache`, and is that intentional?
