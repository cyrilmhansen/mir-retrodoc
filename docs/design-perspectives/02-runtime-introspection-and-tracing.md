# Runtime Introspection And Tracing

## Tracing Goals

Hypothesis: In this design perspective, tracing is primarily for observability, not just for deciding whether code should be compiled.

Runtime facts worth collecting:

- function call counts;
- basic-block entry counts;
- edge and branch frequencies;
- execution time by function or region;
- allocation behavior;
- code-cache allocation and patching behavior;
- instrumentation overhead;
- type or value observations when relevant;
- replacement opportunities and replacement history.

Fact: MIR already has whole-function, lazy-function, and lazy-BB execution paths, but the inspected code does not expose a structured runtime tracing API for these metrics.

## Observability Model

Hypothesis: An observable runtime should be able to answer:

- what code exists;
- when it was compiled;
- where it lives in the code cache;
- how often it executed;
- whether it was replaced;
- why replacement was chosen;
- what overhead instrumentation added.

Fact: MIR stores generated function body addresses in `func_item->u.func->machine_code` and call targets in `func_item->u.func->call_addr` during whole-function generation (`mir-gen.c:9474-9502`). Code cache holders track page ranges through `start`, `free`, and `bound` (`mir.c:4353-4364`).

Inference: MIR has enough low-level address information to inspire code-range metadata, but not a complete introspection model.

## Instrumentation Levels

Hypothesis: Instrumentation should be staged:

- Level 0: no runtime counters; only static metadata.
- Level 1: function entry counts and compile/code-cache events.
- Level 2: basic-block entry counts.
- Level 3: edge/branch counts and coarse timing.
- Level 4: value/type observations and allocation traces.
- Level 5: replacement diagnostics and differential checks against interpreter/oracle mode.

Inference: Function-level counters are cheaper than block or edge counters because they can be attached to thunks, function prologues, or call wrappers. Edge counters need branch-site cooperation or block-entry attribution.

Question: Should instrumentation be inserted into generated code, maintained by wrappers, sampled externally, or collected by a side-channel in an emulator/fantasy runtime?

## Function-Level Metrics

Useful metrics:

- call count;
- cumulative time;
- last call timestamp;
- compiled address;
- code size;
- replacement generation number;
- fallback/interpreter count.

Fact: `MIR_gen` has access to `code_len` and `machine_code` when publishing a function (`mir-gen.c:9474-9485`). Debug output can print code generation time and code size (`mir-gen.c:9491-9498`).

Inference: Function-level code-size and compile-time metrics are close to existing generator state, but they are currently printed under debug rather than stored in a public runtime data structure.

## Counter Semantics

Question: Does a function call counter count stable-thunk invocations?

Question: Does it count physical entry into a specific generated code version?

Question: Does it count logical function calls after direct-call rewriting?

Question: How are recursive calls counted?

Question: How are inlined calls represented?

Question: How are replacement versions merged or separated?

Hypothesis: For MIR-F0, define two concepts even if only one is implemented initially:

- `logical_calls`: calls to the function as an IR-level entity.
- `code_entries`: entries into a specific compiled code version.

Hypothesis: If prototype 1 cannot support both, choose `logical_calls` and document that direct-call rewriting, inlining, and replacement are out of scope.

## Basic-Block-Level Metrics

Useful metrics:

- block entry count;
- first/last execution;
- generated code address for block version;
- fall-through count;
- replacement count.

Fact: Lazy BB mode creates `bb_stub` records over instruction ranges and `bb_version` records with generated addresses and target data (`mir-gen.c:337-355`). Basic-block versions are generated by `generate_bb_version_machine_code` (`mir-gen.c:9807-9994`).

Inference: Lazy BB machinery is useful as a source reference for block identity and block code publication, but it should not be assumed to be the correct tracing architecture.

## Edge/Branch Metrics

Useful metrics:

- per-branch taken/not-taken counts;
- switch case counts;
- indirect jump targets;
- exception/unusual control-flow counts if supported.

Fact: MIR generator CFG construction creates edges between basic blocks during `build_func_cfg` (`mir-gen.c:1571-1887`). The generator's CFG is destroyed after generation (`mir-gen.c:1889-1912`).

Inference: A future observable runtime would need persistent block/edge IDs if it wants branch data after compilation. MIR's generator CFG is a temporary compiler structure, not a retained runtime graph.

## Code-Cache Metrics

Useful metrics:

- mapped bytes;
- published bytes;
- free bytes per holder;
- patch count;
- instruction-cache flush count;
- protection-toggle count;
- per-function or per-region code range.

Fact: Code holders track `start`, `free`, and `bound` (`mir.c:4353-4364`). Publication and patching go through `_MIR_publish_code`, `_MIR_change_code`, `_MIR_update_code_arr`, and `_MIR_update_code` (`mir.c:4426-4483`).

Inference: Code-cache metrics could be collected centrally by wrapping these publication/patching calls in a future system.

## Runtime Overhead Trade-Offs

Inference: Instrumentation in every block or edge can dominate execution cost in small functions. Sampling reduces overhead but gives less deterministic data. Wrapper-based function counters are cheaper but miss intra-function behavior.

Hypothesis: A baseline runtime should make tracing levels explicit and record their estimated overhead, instead of silently changing generated code quality.

Question: Can a fantasy-computer runtime expose counters through emulator support rather than inserting counter instructions?

## Tooling Interface Ideas

Hypothesis: A useful tooling API would expose:

- module/function list;
- IR-to-code mapping;
- code cache ranges;
- counters by function/block/edge;
- replacement history;
- current instrumentation level;
- interpreter/oracle comparison results;
- exportable trace snapshots.

Fact: MIR has public item/module structures and output functions, but the inspected JIT/runtime APIs do not provide a stable structured trace snapshot interface.

## First Trace Snapshot Format

Hypothesis: The first trace snapshot should be stable enough for regression tests, even if it is not a MIR API.

Prototype JSON-like schema:

```json
{
  "schema": "mir-f0-trace-snapshot-v0",
  "module_id": "...",
  "run_id": "...",
  "functions": [
    {
      "function_id": "...",
      "name": "...",
      "code_versions": [
        {
          "version_id": 0,
          "kind": "baseline",
          "address": "...",
          "size": 0,
          "active": true
        }
      ],
      "counters": {
        "calls": 0
      },
      "blocks": [
        {
          "block_id": 0,
          "entry_count": 0
        }
      ]
    }
  ],
  "code_cache": {
    "mapped_bytes": 0,
    "published_bytes": 0,
    "patch_count": 0
  }
}
```

Inference: Addresses may be unavailable or symbolic in an emulator or fantasy runtime.

Rule: This schema is a prototype test artifact, not an upstream MIR API.
